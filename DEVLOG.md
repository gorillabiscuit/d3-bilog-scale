# DEVLOG — d3-outlier-graphs

Decision log for the article. Timestamped one-liners only.

---

- **[14:30]** Rejected uniform hatch lines across entire log tail: it recreates the bunched-lines look at 45° and communicates nothing new about per-region compression.
- **[14:35]** Redesigned hatch texture to per-band density encoding: each equal-dollar-step section gets its own SVG `<pattern>` whose line spacing scales with that band's pixel width relative to the widest band in the tail — widest band = sparsest hatch (8px), narrower = denser, sub-1px = solid fill.
- **[14:40]** Key insight that forced relative normalisation: absolute normalisation against linearW caused all bands to fall below the solid-fill threshold on datasets with highly-compressed tails (NYC property data: first right-tail band only 11.5px vs 113px linear window, giving 1.2px spacing before any pattern renders).

- **[13:50]** Root cause found for all the left/right hatch divergence: commit 0cbede2 (Jun 20) silently swapped both tail sub-scales from scalePow(0.3) to scaleLog, bundled into a ruler-ticks/annotations commit with no DEVLOG entry. But scalePow(0.3) was never a mirror either — exponent<1 expands the small end exactly like log. A plain log/pow tail CANNOT mirror its opposite because log measures ratio, which is unbounded going up but floors at zero going down; its slope 1/x always expands the small extreme. All the collectLeftBands / index-formula / gap-fill patches were the hatch layer fighting a scale that was never symmetric.
- **[14:00]** Replaced both tail sub-scales with d3.scaleSymlog fed DISTANCE OUTWARD from the linear-window boundary ((x−xHi) right, (xLo−x) left). Distance is symmetric (no zero-floor), so symlog compresses both tails identically — small-value outliers bunch at the left edge exactly as large-value outliers bunch at the right. This is the windowed generalisation of symlog (linear middle via the existing drag-handle window, log tails). Both tails now run one shared `symlogTail()`; deleted collectLeftBands, the index formula, the boundary-gap fill, and the logMin clamp (symlog handles $0/negatives natively). drag-invert proxies switched scaleLog→scaleSymlog too — stable on USGS (1→251M), no exponential blowup.
- **[14:05]** Chose a span-relative symlog constant (span/κ, κ=12, identical both tails) over slope-matching the constant to the window. Slope-matching was kink-free but tied each tail's compression to its own data span, so the wider tail (NYC right $156M vs left $4.54M) compressed far harder — not a visual mirror. Fixed κ gives both tails one compression profile: pixel-identical hatch bands ([24.31, 11.3, 7.44] both sides on NYC and USGS) regardless of dollar range. Trade-off: a slope kink at each boundary, which actually reinforces the visible log|linear|log region distinction the annotations already label.
- **[14:20]** Reverted the fixed-κ constant — it was wrong. User clarified the goal is faithful inspection of extreme data, NOT a forced visual mirror: "I don't want to massage how that works depending on the data… my ultimate goal is not a visual outcome." Fixed κ normalised away the data (pixel-identical tails regardless of range) = exactly the massaging to avoid. Restored the slope-matched constant: same law both tails (boundary slope = window slope), but the constant is *derived* from each tail's own data+geometry, so output follows the data — NYC now renders honestly asymmetric (left near-linear 5 even bands as its $4.54M range barely needs compression; right one band then solid as its $156M range crams hard). True symmetry now emerges only when the data is symmetric (equal range either side of a centred window), never imposed.
- **[14:40]** Replaced the discrete per-band hatch with a continuous scale-driven fill. User feedback: a near-linear tail (range ≈ linear window) still showed ~5 separate hatch bands, "inferring five different logs, which is obviously incorrect" — the fixed N-band model invents structure a single continuous symlog doesn't have. Deleted collectBands, drawTailBands, BAND_MIN_PX, TAIL_BANDS. Verified continuous (perBandClips=0) on NYC and panned USGS; 124 tests pass.

- **[16:00]** New branch exp/log-rate-arrows: replace the hatch entirely with a "tail ruler" annotation. The linear window's dollar width W is the unit; each log tail is tiled into W-chunks, each = 1× the linear distance. A full-height post marks every chunk boundary (the same vertical line the linear section has at its edge); a chunk wide enough also gets a ←→ arrow, and one wide enough for text gets a stacked "log / ×1" label (matching the linear annotation's type-over-value layout). Each log region is tinted a touch lighter than the linear section so they read as distinct.
- **[16:05]** Density cap is the codebase's canonical tail-tick pattern: walk W-chunks outward, break as soon as a chunk is < RULER_MIN_PX (2px) — monotonic symlog compression guarantees every later chunk is narrower, so this bounds posts at ~(tail width / 2px), never thousands. Past the point where individual arrows no longer fit, the remaining chunks collapse into ONE arrow labelled "log ×N" (N from the scale: remaining dollars ÷ W) — e.g. ×17, ×1.5M, or fractional ×0.5 when a whole tail is smaller than the window. Reuses the overflow-indicator / broken-axis convention.
